# OpenClaw Cortex Memory：P0 + P1 + P2 改造开发与落地测试方案

## 1. 目标与范围

本方案用于将 `openclaw-cortex-memory` 升级为可长期稳定运行、可持续发布、可在 OpenClaw 3/22+ 生态中平滑安装与调用的插件版本。  
覆盖范围：

- 安装来源与安装记录链路（ClawHub-first + npm fallback 时代）
- 插件与主 Agent 的接口兼容（tool/hook/register API 形态）
- 运行时稳定性（配置路径、状态路径、回退链路、会话链路）
- 大模型调用链路审计（Embedding / LLM / Reranker 的真实接入现状与后续落地）

## 2. 现状审计结论（基线）

### 2.1 当前已具备能力

- 已实现插件主入口与工具注册、Hook 注册、启停与回退控制。
- 对 OpenClaw API 做了兼容层处理：
  - `registerTool` 同时支持 `execute` 与 `handler` 两种调用形态。
  - Hook 同时兼容 `api.on(...)` 与 `api.registerHook(...)`。
- 构建产物完整，`dist/index.js` 与 `dist/openclaw.plugin.json` 可正常打包。

### 2.2 当前关键风险

- 安装来源在 ClawHub-first 机制下存在“裸包名解析不确定性”风险（文档仍偏向裸包名描述）。
- 配置/状态路径解析未统一覆盖 `OPENCLAW_CONFIG_PATH`、`OPENCLAW_STATE_DIR` 等优先级。
- 文档中“本地源码安装自动构建”的描述与当前 `package.json` 脚本不一致（需补 `prepare` 自动编译）。
- `openclaw.plugin.json` 与 `package.json` 版本号不一致，影响排障与发布一致性。
- **大模型调用现状：当前 TS 引擎尚未具备可验证的外部 Embedding/LLM/Reranker 回归门禁**，需要补齐自动化测试闭环。

## 3. P0 改造（必须先落地）

### P0-1：安装来源确定性改造

- 文档与发布规范统一为“显式来源策略”：
  - 生产推荐：优先 ClawHub 规范源（上线后）
  - 回退策略：npm 安装源保底
- 所有安装文档明确“禁止仅依赖裸包名的隐式来源期望”。

**验收标准**

- 安装文档在 ClawHub 有包与无包两种场景下都有明确命令。
- 用户可据文档准确判断安装来源，不再依赖隐式解析。

### P0-2：配置与状态路径优先级统一

- 统一路径解析优先级（运行时与脚本一致）：
  1. `OPENCLAW_CONFIG_PATH`
  2. `OPENCLAW_STATE_DIR`
  3. `OPENCLAW_BASE_PATH`
  4. 默认 `~/.openclaw`
- 覆盖文件：
  - 插件入口配置定位
  - session 同步定位
  - CLI/卸载脚本配置定位

**验收标准**

- 在以上 4 类路径输入下均能正确读取配置与会话目录。
- 迁移环境（自定义 state/config）不再出现“读错目录”。

### P0-3：版本一致性与发布一致性

- 强制 `package.json` 与 `openclaw.plugin.json` 版本一致。
- 发布前门禁新增版本一致性检查。

**验收标准**

- `npm pack --dry-run` 与插件运行时展示版本一致。

## 4. P1 改造（上线前必须完成）

### P1-1：主 Agent 兼容调用稳定性增强

- 固化兼容矩阵（必须保持通过）：
  - Tool：`execute({args,context})`
  - Tool：`handler(args, context)`
  - Hook：`api.on(event, handler)`
  - Hook：`api.registerHook({event, handler})`
- 增加回归脚本，验证工具注册、Hook 注册、回退工具注册。

**验收标准**

- 上述 4 类调用形态均通过回归测试。
- `disable()` 后 fallback 工具可用（`search_memory`/`store_event`/`cortex_memory_status`）。

### P1-2：卸载与清理一致性

- 卸载时除 `plugins.entries` 外，补充清理 `plugins.allow` 中插件条目（保留可选开关策略）。
- 补充“保留数据 / 保留配置”路径的明确行为定义与验证。

**验收标准**

- 卸载后不会留下误导性配置残留。
- `--keep-data`、`--keep-config` 行为可预测且可复现。

### P1-3：运行观测与失败定位

- 统一关键日志事件：
  - 注册、启用、禁用、回退、反注册
  - Hook 执行超时与跳过
  - Sync/SessionEnd 的导入与跳过统计
- 输出可检索的结构化关键信息，便于线上排障。

**验收标准**

- 常见故障（配置错、路径错、禁用状态、回退触发）可从日志快速定位。

## 5. P2 改造（增强项，建议排期）

### P2-1：大模型调用真实落地（Embedding/LLM/Reranker）

当前状态是“配置字段齐全但并未真实调用模型 API”。  
P2 建议落地分层：

- Embedding 层：向量化写入 + 语义检索（替换当前纯文本打分）
- Reranker 层：二次排序提高召回精度
- LLM 层：反思/规则抽取从模板化文本升级为模型抽取

**验收标准**

- 在配置完整时触发真实外部模型调用，且具备 timeout/retry/错误分级。
- 在模型不可用时保留当前本地降级路径（服务不中断）。

### P2-2：ClawHub 发布与版本治理

- 建立 ClawHub 发布流程、兼容性元数据、升级策略。
- 实施“主线稳定版 + 预发布版”分支策略。

**验收标准**

- 用户通过 ClawHub-first 安装可稳定获取兼容版本。

### P2-3：生产门禁流水线

- 构建门禁：`npm run build`
- 打包门禁：`npm pack --dry-run`
- 兼容门禁：mock host API 双形态回归
- 回退门禁：disable 后 fallback 工具回归

**验收标准**

- 任一门禁失败即禁止发版。
